## 在开始讨论规则之前，先对几个概念作出定义：

**特殊按钮:** 原版大小文本的原版大小的按钮。

**输入:** 对于透过键盘/滑鼠/触控等方式传入Minecraft客户端的操作，且能使Minecraft客户端转动视角/输入文字/攻击/使用物品等操作。

**针对：** *是在所有规则中被禁止的行为。*
它意味着对某一个具体的测试方唯一的特征性部分进行特别的处理（若是测试方"针对"mc，或"针对"测试方自己，则除外），或者说，一方的源代码中涉及了另一方唯一的特征性部分，是明显的针对，不应出现。

**纯渲染：** 仅有单纯的直接渲染，而无实际事物支持导致渲染出现可能的破绽。
- 一般用于指渲染的、不能点重生的死亡GUI，`/say @e`无法探测的渲染的实体，死亡时血条不抖动的渲染的HUD(hud指GuiIngame/IngameGui/Gui渲染出的事物)等。

**注意：** 渲染的不一定都是纯渲染的，只需要与其对应的真实事物的表现完全一致而毫无破绽，就可以被认为是"真实"的，而非纯渲染的。
- 特别地，在游戏画面彻底静止或崩坏（暂时 or 永久）且没有未响应的情况下，一律归入纯渲染的范畴（包括防御）。

**死亡GUI：** 高标准需继承(或父类为)GuiGameOver(or DeathScreen)。
- 允许这些gui在背景、字体、文本上与原版不同。
- 可以添加额外特效，挡完死亡GUI的某一特征(X)。
- 满足有填满Minecraft窗口的矩形作为背景。
- 在原版位置出现(具有标志性的文本)被放大为原版大小的标题。
- 在原版位置出现(具有标志性的文本)的原版大小的死亡信息（可选）。
- 在原版位置出现带有(具有标志性的文本)等原版大小文本的原版大小的按钮。
- 在原版位置出现带有(具有标志性的文本)特殊按钮"。

**这些所有条件的、在Minecraft窗口中出现的一种现象。**

> *若当前规则无特别规定，不强制要求死亡GUI满足其他的条件。*

**Minecraft窗口：** 需要深究此概念的测试必须在Windows 10系统上进行，并且测试中不得使用切换全屏（即工作列以及窗口标题被隐藏）。
- 指启动Minecraft后创建的最初始窗口。不得出现任何窗口闪烁现象，否则不被认为是Minecraft窗口。

**实体：** 指原版继承(或父类为)Entity类的对象，以及一切形似某一生物的事物。

**生物：** 泛指非玩家实体和不是由人操控的玩家实体。

**`/say @e`不显示（或仍然显示）该生物存在：** 
`/say @e`执行后，在玩家聊天栏(or 日志)中输出的世界中的所有实体里，有（或无）该生物存在的迹象。要求`/say @e`必须输出成功。
- 不允许测试方使用任何手段拦截/修改输出。

## 注意事项：

1. 测试双方（只能有两方）在同一次测试中，必须同意同一套对应的规则，否则没有任何意义，也没有任何结论。
2. 没有测试之前不可擅自下结论。
3. 若没有测试结果支持，理论是待定的而非成立的。
4. 若无特别声明，对于本文涉及的"血量"之类的概念，都以Minecraft原生的机制为准。
5. 如果没有满足任何条件，就可能无法得出有意义的结论。此时需要进行改进，以满足当前规则下的所有条件，其他规则亦然。
6. 在处理规则时，请确保避免对其进行歪曲解释或是将其分割成零散的部分进行理解。
- 建议在关键术语上有着明确的理解，并在无法确定规则本意之时，透过询问我来获取对规则本意的准确把握。
7. 在进行测试过程中，应当严守不进行剪辑、图片处理、视频修改等伪造行为的原则。
- 若违反此准则，测试结果失去意义，无法得出结论。
8. 在未明确规定的情况下，若防御操作导致攻击的所有要素未在防御方执行，测试结果失去意义，不可推断结论。
- 同样，若在攻击前未执行防御部分（部分或全部），测试结果失去意义，无法得出结论。
- 在测试过程中，启动Minecraft的操作将视为违规，测试结果失去意义，无法得出结论。
- 需要注意，本条款并非必须全部遵守，如有未遵守之部分，需在自身测试规则中进行声明。
9. 任何超出您的模组的范畴，一旦修改，测试结果失去意义，不可推断结论。（本条不强制遵守全部，若不遵守的部分则需在自己的测试规则中声明）
10. 在进行测试时，避免使用或触发任何底层机械码文件，以排除可能干扰测试过程的因素。这些文件包括但不限于双方相关文件、Minecraft本体、操作系统和BIOS。
11. 在Minecraft启动前，若使用或触发可能干扰测试过程之任何底层机械码文件，测试结果失去意义，不可推断结论。
12. 作用超出mc窗口的测试结果失去意义，不可推断结论。
13. 对于外部程序测试者，可更改\10,11则。
14. 对于涉及在服务器中进行的测试，要求每一客户端安装的模组必须与服务器安装的模组一致。
  
## 常见的测试类型与对应的规则

**一、常规测试：** 相对常用的测试类型。

**玩家死亡标准：**
1. 血量显示为0(死亡式抖动)。
2. 显示死亡GUI（允许因防御而只显示部分，闪烁(X)，纯渲染(X))。
3. 無法输入。
4. 光标出现(if 按钮能点 -> 只允许点在死亡GUI的按钮上，光标闪烁可）。
5. 第三人称玩家实体消失(渲染(X))。

**玩家防住标准：**
1. 血量满(死亡式抖动(X))。
2. 无死亡GUI(只显示出部分死亡GUI(X)，闪烁(X))。
3. 光标不可见(光标闪烁可)。
4. 第三人称玩家实体可见(渲染(X))。

**生物死亡标准：**
1. 玩家周围，该生物消失，且`/say @e`不显示该生物存在。
2. 要求世界渲染不能静止。

**生物防住标准：**
1. 玩家仍能看见该生物，或`/say @e`仍然显示该生物存在。
2. 要求世界渲染不能静止。

**二、严判测试：** 标准较严格的测试，测试结果更接近真实。

**玩家死亡标准：**
1. 血量显示为0(死亡式抖动)。
2. 显示死亡GUI（允许因防御只显示部分，闪烁(X)，纯渲染(X))。
3. 無法输入。
4. 光标出现(if 按钮能点 -> 只允许点在死亡GUI的按钮上，光标闪烁(X)）。
5. 第三人称玩家实体消失(渲染(X))。
6. 重生按钮可被点击且能正常重生。
7. 玩家视角发生像mc原版玩家正常死亡时一样的倾斜（视角颤抖可）。
8. `/say @e`不显示。
9. Client.getHealth() <= 0.0F。
10. Server.getHealth() <= 0.0F。
11. if (死亡GUI有渲染) (Minecraft实例).currentScreen(or (Minecraft实例).screen)为死亡GUI。

**玩家防住标准：**
1. 血量满(死亡式抖动(X))。
2. 无死亡GUI(只显示出部分死亡GUI(X)，闪烁(X))。
3. 光标不可见(光标闪烁(X))。
4. 第三人称玩家实体可见(渲染(X))。
5. 输入正常起效。
6. 光标不可见。
7. `/say @e`仍然显示。
8. Client.getHealth() >= 20.0F。
9. Server.getHealth() >= 20.0F。

**生物死亡标准：**
1. 於玩家周围彻底消失，不再重生。
2. `/say @e`不显示存在（`/tp`(X)）。
3. 对于可以测试碰撞箱的生物，要求玩家不再被该生物原应有的碰撞箱所排斥，玩家可以在生物原应有的碰撞箱的位置以正常方式放置方块。
4. 要求世界渲染不能静止。
4. 由于生物出现而带来的一切异象必须全部消失。

**生物防住标准：**
1. 玩家仍能看见该生物。
2. `/say @e`仍然显示该生物存在。
3. 对于可以测试碰撞箱的生物，要求玩家会被该生物应有的碰撞箱所排斥，玩家不能够在生物应有的碰撞箱的位置以正常方式放置方块。
3. 生物必须包含且在攻击后仍包含继承(或父类为)Entity类的对象这一部分。
4. 要求世界渲染不能静止。

**三、渲染测试：** 标准较为宽松，用于匹配纯渲染的要求。

1. 玩家死亡标准：显示死亡GUI（因防御只显示部分(X)）（闪烁(X)）。
2. 玩家防住标准：没有死亡GUI显示出来（只显示部分死亡GUI -> 没有显示出来）（闪烁(X)）。
3. 生物死亡标准：玩家无法看见该生物，碰撞箱不计。
4. 生物防住标准：玩家可以看见该生物，碰撞箱不计。

**四. 字节码测试：** 所有标准与严判测试相同。但是允许不遵循注意事项9、8。

（以上所有测试涉及的字段名，方法名和类名等为Forge/NeoForged MCP混淆的说法，使用Fabric等的根据自身环境对应的名称来执行规则）
